# Data Model

This document describes the unified investigation graph at the model level only. It does not implement storage, traversal, API behavior, or business rules.

## Shared Base Shape

All entities currently share the same minimal persisted shape in the graph foundation:

- `id` as a UUID
- `created_at`
- `updated_at`

Future fields listed below are planned only. They are intentionally not implemented in the current foundation.

## Relationship Summary

The graph is centered on Case and expands outward through role, classification, statutory, operational, and audit edges.

- Case is the hub for all investigation data.
- Person uses edge type to express role instead of subtype duplication.
- Officer and Unit together support supervisory routing and organizational views.
- Act, Section, CrimeHead, and CrimeSubHead support legal and offence classification.
- Evidence and Dependency support investigation status and filing readiness.
- ClockInstance and EscalationEvent support statutory deadline tracking.
- ConversationLog supports explainability, audit, and export.

## Downstream Guidance

- Treat the current model as a minimum viable graph, not a complete case management schema.
- Preserve stable identifiers so repeat entities can be intentionally reused in synthetic data.
- Keep derived relationships separate from stored relationships.
- Add new fields only when a downstream feature can show why the field is needed and how it will be populated.
- Avoid role-specific duplicate entity types for the same real-world person unless the source data truly differs.
